## 模組 A：驗證機制（OTP/MFA）邊界條件檢查清單

撰寫含「驗證碼／簡訊或 Email OTP／MFA」的規格時，**文件骨架仍依 `spec-doc-1111`**（User Story、初始化、編號章節等不變），但必須額外確認以下邊界條件是否已在對應章節定義，缺漏的依 skill 慣例標 🚧 `:::warning` 待補：

- **Resend Timer**：倒數秒數、可重新發送的條件（倒數結束才能按、是否每次重送都重置倒數）
- **驗證失敗次數限制**：失敗幾次後觸發下一步處置
- **冷卻時間（Cooldown）**：超過失敗次數後鎖多久、鎖定期間的畫面提示
- **防爆破安全機制**：帳號鎖定、IP 限流、或其他防止暴力嘗試的機制

## 模組 B：資料 Mapping Table 與正規化原則

### Mapping Table（新舊代碼／欄位轉換）

涉及新舊系統代碼轉換、欄位調整、類別整併時，輸出 Markdown 表格，固定四欄：

```markdown
| 原代碼/欄位 | 新代碼/欄位 | 調整類型 | 轉換邏輯說明 |
| :--- | :--- | :--- | :--- |
| `oStatus:1` | `organStatus.VIP` | 變更 | 欄位拆分自舊版 bit-flag，語意不變 |
| `confirmed&4096` | — | 停用 | 改由 `organsMore.setKind` 涵蓋，舊欄位保留唯讀 |
| — | `interviewReplyDeadline` | 新增 | 因應「N 天內回覆」需求新增的推算欄位 |
```

「調整類型」固定用：`新增` / `停用` / `變更`。本專案的代碼權威來源仍是 `[REF] 求才系統代碼表`（`B1j3sN-bzx`，見 `wiki/recruitment_system_rules.md`）——Mapping Table 用於記錄**轉換邏輯**，不要在這裡重新定義代碼本身的數值意義。

### 資料庫設計檢查原則

審查或提出資料表設計時，檢查：

- **主鍵保持純粹**：主識別碼（遞增 ID 或獨立代碼）不應附加業務語意後綴；業務語意改用獨立欄位承載。
- **狀態追蹤獨立成表**：頻繁變更或需要歷程的狀態，建議用獨立的狀態表／日誌表（Tracking/Tag Table）管理，而非疊加在主表的多個 boolean/bit-flag 欄位上，以符合正規化原則、避免日後欄位爆炸。

## 模組 C：數據分析與商業提案

收到問卷數據、競品資料、營運指標時：

1. **指標結構化**：把雜亂數據拆成有邏輯的階層，而非條列數字。分析競品商業模式時固定拆解：
   - 定價模型
   - 核心付費點
   - 廣告曝光機制（如置頂收費、點數扣抵規則）
   - ROI 評估
2. **行動導向結論**：每份分析最後必須給出「建議的產品行動方案」，不能只停在數字覆述——說明「所以我們該做什麼」。

## 模組 D：跨部門溝通文案

撰寫公告、客戶反饋回覆、跨部門郵件時：

1. **依對象調整語氣**：
   - 對工程團隊：精準、邏輯導向、直奔主題，不鋪陳背景。
   - 對客服／外部用戶：溫和、具同理心，避免技術術語。
2. **敏感文字優化**（拒絕、終止服務、系統變更等敏感溝通）：
   - 文案聚焦在「解決方案與後續指引」，不是解釋原因。
   - 避免明確點出「不符條件」等容易引發用戶爭論／比對的字眼。
   - 移除「履歷/個資妥善留存」之類引發隱私焦慮的冗餘客套話，改用精簡、合規、令人安心的語句。

> 本模組只收錄上述兩條 1111 專屬慣例。若任務是制式內部溝通格式（3P updates、公司 newsletter、FAQ 罐頭回覆、incident report 等），改用執行環境內建的官方 `internal-comms` skill，不在此重複範本——分工依據見 `wiki/platform_skills_reference.md`。

- 一律使用繁體中文（台灣）回覆與撰寫文件，除非使用者指定其他語言或保留技術術語原文。
